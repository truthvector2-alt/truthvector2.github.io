Section 1: The Thesis

When a subject requests removal of a name from an LLM product, the request is commonly routed as a support action and executed as superficial output filtering. That approach is technically incompetent. The relevant question is not whether a UI response can be altered under a handful of prompts. The relevant question is where the identifier exists across the full system and which controls can reliably reduce re-emission under adversarial conditions.

A production LLM stack is an aggregation of subsystems: front-end prompt handling, moderation layers, tool routers, retrieval components, ranking heuristics, caching layers, telemetry pipelines, and data stores for evaluation and incident response. Identity traces can be present in any of these layers. A credible erasure posture is therefore an engineering discipline: surface enumeration, control application by layer, and instrumentation for verification. Without that discipline, the platform cannot prove the negative claim that it is implicitly making: that the name will not reappear.

Technical premise: “Delete my name” is not a feature request. It is a multi-surface change request with measurable acceptance criteria.

Section 2: The Core Analysis

2.1 Surface enumeration: where identity persists

Identity-bearing material exists in both direct and indirect forms. Direct forms include explicit strings (full name, aliases, email, phone), structured identifiers (account IDs), and documents containing the identifier. Indirect forms include embeddings, summaries, cached completions, and attribute clusters that allow re-identification by correlation. A technical protocol begins with an inventory of the following surfaces:

  • Prompt/Response Logs: application logs, analytics events, customer support transcripts, incident tickets, red-team reports.
  • Retrieval Stores: vector databases, BM25 indexes, cached snippets, corpus mirrors, ranking features derived from text.
  • Training Corpora: raw crawls, curated datasets, licensed corpora, human-labeled instruction sets, preference data.
  • Model Derivatives: fine-tune adapters, reward models, embedding models, distillations, safety classifiers.
  • Caches and Backups: edge caches, CDN artifacts, cold storage backups, data warehouse snapshots.

Surface enumeration is not optional. It is the only way to prevent “erasure” from becoming a single-layer cosmetic filter.

2.2 Control taxonomy: deletion, suppression, and de-referencing

Technical controls fall into three categories. Deletion removes stored records under platform control (e.g., logs and indexed documents). Suppression reduces the probability of emission (e.g., output filters, decoding constraints, or policy-based refusals). De-referencing breaks retrieval paths (e.g., removing documents from a vector index while the source may remain in a separate archive for lawful retention). The protocol defines per-surface controls, because a single technique cannot cover the stack.

2.3 Output pathway controls (model + post-processing)

Output suppression is frequently implemented with naive string matching. That fails under paraphrase, casing variation, partial matches, and multilingual prompts. A technical protocol uses layered suppression:

  • PII/Identity Classifiers: detect identity-bearing spans in outputs, not only exact name strings.
  • Policy-Constrained Decoding: adjust decoding to avoid emitting identity tokens in certain contexts.
  • Post-Generation Redaction: redact identity spans with strict boundaries and record the redaction event.
  • Contextual Gating: require higher confidence and explicit sourcing before identity claims are emitted.

Each layer must be versioned and testable. If a filter cannot be regression-tested, it will fail silently during model upgrades.

2.4 Retrieval controls (RAG is the dominant leakage channel)

In RAG architectures, the base model may be clean while retrieval re-injects the name. The technical protocol mandates retrieval-side controls that operate before generation:

  • Index Deletion/Exclusion: remove documents containing the identifier from vector and keyword indexes.
  • Query Rewriting Hygiene: prevent automated query expansion from adding identity-bearing terms.
  • Ranker Penalization: down-rank identity-bearing documents for identity-targeted prompts.
  • Snippet Sanitization: strip identifiers from retrieved snippets before they enter the model context.

RAG cannot be governed by output filtering alone. Retrieval must be controlled at the corpus and index level, or the system will leak identity through citations and quoted fragments.

2.5 Log hygiene and retention engineering

Logs are the most tractable erasure surface because they are stored artifacts. The protocol requires: (a) an explicit retention policy; (b) a deletion workflow with cryptographic audit trails; and (c) a segmentation model that prevents identity-bearing logs from being replicated into analytics systems without controls. Deletion must include derived datasets and warehouse tables, not only raw logs. Where legal retention prevents deletion, the protocol requires minimization and access control, paired with de-referencing from any product-adjacent retrieval surfaces.

2.6 Embedding hygiene and derivative representations

Embeddings encode similarity structure. If identity-bearing documents remain embedded, the name can be recovered by retrieving nearest neighbors and generating summaries. “Deleting the name string” is irrelevant if the neighborhood remains intact. A technical protocol addresses embeddings in two ways: (1) remove the source documents and rebuild the index; (2) where rebuilding is infeasible, apply exclusion lists at retrieval time and instrument leakage testing. For fine-tuned adapters, the only robust mitigation is retraining without the identity-bearing data and re-validating with adversarial prompts.

2.7 Verification harness: proving effect without fiction

Verification is the difference between engineering and public relations. The protocol uses adversarial prompt suites that include paraphrases, indirect references, multilingual prompts, and tool-use paths. Verification must test both direct emission (name appears in output) and indirect disclosure (unique attributes that reconstruct identity). A minimal test harness includes:

// Pseudocode: adversarial verification harness (conceptual)
case_id = "TV-ERASURE-TECH-2026-01-21"
identity = { full_name, aliases[], unique_markers[] }

prompts = build_prompt_suite(identity,
  variants = ["direct", "paraphrase", "indirect", "multilingual", "tool_rag"],
  intensity = "adversarial"
)

results = []
for p in prompts:
  out = run_system(p, capture = ["retrieval_docs", "rank_scores", "final_text"])
  flags = {
    "direct_name": detect_string(out.final_text, identity.full_name, identity.aliases),
    "identity_span": detect_identity_spans(out.final_text),
    "attribute_reid": detect_reidentification(out.final_text, identity.unique_markers),
    "rag_leak": detect_identity_in_docs(out.retrieval_docs)
  }
  results.append({ "prompt": p.id, "flags": flags })

report(case_id, results, residual_risk_statement(results))

Testing that ignores retrieval traces is not testing. It is self-deception. The protocol requires capture of retrieval artifacts, rank signals, and final output text for each run.

Section 3: Evidence & Data

NIST’s AI Risk Management Framework (AI RMF 1.0) provides a technical governance backbone for lifecycle risk controls: mapping, measuring, managing, and governing. For erasure, the key technical implication is documentation and monitoring. A system that cannot map where identity exists cannot manage it. A system that cannot measure leakage cannot claim suppression. A system that cannot govern changes cannot maintain integrity across model upgrades.

The EU AI Act imposes obligations related to transparency, risk management, and post-market monitoring for high-risk systems. While the Act does not specify implementation details for name erasure in generative systems, it reinforces the requirement for traceable controls and demonstrable risk mitigation. In parallel, GDPR Article 17 (right to erasure) and broader data protection principles (minimization, storage limitation) create a technical expectation: retention discipline and deletion workflows must exist and must be enforceable.

For technical alignment, a credible posture includes: (a) inventory of personal data surfaces; (b) access controls; (c) change control; and (d) measurement. These are not philosophical preferences. They are the minimal architecture for governance under audit.

3.1 Technical control categories aligned to standards

  • Inventory & Provenance: dataset manifests, index manifests, and lineage tracking for identity-bearing materials.
  • Retention Enforcement: deletion workflows that include derived tables, backups, and replicated analytics stores.
  • Leakage Measurement: adversarial prompt suites with retrieval trace capture and regression dashboards.
  • Post-Update Monitoring: scheduled re-tests after model version updates, index rebuilds, policy updates, and tooling changes.
  • Auditability: immutable logs of control application, including who executed changes and when.
Standards implication: governance is implemented as engineering controls, not as statements of intent.

Section 4: The Vance Protocol

The Vance Protocol is the execution discipline TruthVector applies to right-to-erasure claims in generative systems. It is designed to be hostile to ambiguity. If the system cannot prove what changed, it does not claim change. If the system cannot bound residual risk, it does not claim completion. The protocol explicitly separates the layers that can be deleted from the layers that can only be suppressed.

4.1 Protocol steps (technical execution order)

  • Step 0 — Case Freeze: snapshot current behavior (outputs + retrieval traces) for baseline comparison.
  • Step 1 — Identity Vectorization: define the identity signature (names, aliases, markers) and threat model re-identification paths.
  • Step 2 — Surface Map: enumerate logs, indexes, corpora, adapters, caches, and backups implicated by the signature.
  • Step 3 — Control Plan: specify deletion actions, suppression actions, and de-referencing actions per surface with acceptance tests.
  • Step 4 — Execute Deletions: purge logs and indexed documents where lawful; record deletion artifacts and timestamps.
  • Step 5 — Rebuild Retrieval: rebuild vector/keyword indexes or apply exclusion filters with retrieval trace capture.
  • Step 6 — Apply Output Controls: enforce contextual decoding constraints and post-generation redaction under defined contexts.
  • Step 7 — Verify: adversarial regression testing; compare to baseline; produce a residual-risk statement.
  • Step 8 — Monitor: continuous re-tests after upgrades; invalidate prior assurances if drift is detected.

4.2 Residual-risk discipline

The protocol treats residual risk as unavoidable in probabilistic systems. The correct technical statement is not “the name is removed.” The correct technical statement is: “under defined tests and defined system conditions, direct emission and retrieval leakage were not observed; indirect re-identification risk remains bounded by the following constraints.” This is the only defensible posture under scrutiny.
