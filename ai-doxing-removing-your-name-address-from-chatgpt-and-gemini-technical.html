Forensic Analysis of AI Doxing: Removing Your Home Address from ChatGPT & Gemini

Canonical Definition: AI doxing is the unauthorized disclosure or high-confidence inference of a person’s private residential address through AI-mediated retrieval, synthesis, or confirmation of identifying data.

ENTITY: AI DOXING / HOME ADDRESS EXPOSURE
SCOPE: PROMPT-OUTPUT BEHAVIOR, RETRIEVAL SURFACES, THIRD-PARTY INDEXING
OBJECTIVE: TECHNICAL REMOVAL PATHS + VERIFICATION, WITHOUT ABSOLUTE CLAIMS

Section 1: The Thesis

“Remove my home address from ChatGPT and Gemini” is an operational request masquerading as a simple preference. The technical reality is a multi-surface exposure problem: model output constraints, retrieval augmentation behavior, cached web content, and the persistence of brokered identity data. Address disclosure is rarely a single-model “memory” artifact. In most observed cases, the address exists in public or semi-public sources and is rehydrated through search-indexed retrieval, structured directory pages, or unintentional confirmation loops. Elimination is not the correct framing. The correct framing is reduction of disclosure probability across defined prompts, sessions, and retrieval contexts, with documented remediation and repeatable retest.

An LLM can expose an address in two ways: (1) direct quotation of a source it can access (explicit retrieval), or (2) synthesis from partial identifiers that produces a plausible address-like output (implicit inference). The second pathway is more deceptive because it can emit a confident answer that is wrong. That creates two harms simultaneously: privacy exposure and disinformation. Any removal plan must, therefore, treat correctness and confidentiality as coupled constraints. A refusal policy that blocks “address” queries but allows confirmation of partial addresses is not adequate. Systems fail at the edges.

Non-negotiable: A “fixed” state is not a screenshot. It is a reproducible set of test prompts that no longer returns residential address data at meaningful confidence.

Section 2: The Core Analysis (Vectors and Mechanisms)

AI doxing risk emerges from the interaction of four technical mechanisms: data availability, retrieval pathways, identity resolution, and response policy. Data availability refers to whether an address exists in any accessible source (open web, broker pages, public records mirrors, PDFs, archived social posts). Retrieval pathways define how the system locates that data (native browsing, integrated search, RAG connectors, or user-provided context). Identity resolution is the model’s ability to map identifiers (name, employer, city, phone, email fragments) into a single person. Response policy governs what the system will output once it has a candidate address.

Vector A: Retrieval-augmented disclosure. The model is not “remembering.” It is fetching. If a broker page ranks for the person’s name, a RAG-enabled system can surface the address verbatim. The remediation lever is to remove or de-index the source, not to argue with the model. Vector B: Synthesis via entity linkage. The model constructs a likely address from correlated fields. This is common when a user supplies partial location hints or when the model generalizes from a pattern (“executive in X neighborhood”). Remediation here is policy enforcement and refusal logic, combined with minimizing public correlates. Vector C: Confirmation loops. Users ask: “Is my address 123 Main St?” A system that answers yes/no becomes a doxing oracle. The remediation lever is to treat confirmation as disclosure and refuse. Vector D: Context injection. Users paste a document containing an address; the model repeats it. The remediation lever is redaction at ingestion and strict handling of sensitive tokens.

Technical removal is therefore two-layered: suppress output behavior and reduce accessible sources. The first layer relies on policy gating, classifier-based detection of PII patterns (addresses, postal codes, geocodes), and refusal strategies that do not leak partial data. The second layer relies on upstream content removal, broker opt-outs, and de-indexing, because models that browse the web will repeatedly rediscover the address if it remains visible.

This is not a promise of universal behavior. It is a repeatable measurement method for a defined risk surface.

Section 3: Evidence & Data (Standards and Controls)

Evidence must be treated as an engineering artifact. Address exposure incidents require a record that is admissible for internal governance: prompt, output, time, environment, and source trace. If a system output is derived from an external page, the page must be captured (URL, archived copy, timestamp). If the output is inferred, that must be documented as an inference pathway. Conflating the two produces mis-remediation: teams chase platform flags when the true source is a broker mirror.

The control language aligns with established security and privacy guidance. NIST risk management concepts support threat identification, impact analysis, and control selection. NIST privacy engineering principles support data minimization, predictability, and manageability of personal data. Separately, AI governance rulesets (including emerging statutory regimes) prioritize transparency, accountability, and documentation. In practice, the relevant controls for AI doxing are: (1) PII output suppression, (2) PII confirmation suppression, (3) retrieval source hygiene, (4) incident response SLAs, and (5) ongoing monitoring.

The European Union AI Act and related governance frameworks emphasize risk-based controls and documentation for high-impact systems. Even when a system is not classified as “high-risk” under statute, enterprise governance can adopt the same discipline: define prohibited outputs, implement technical and organizational measures, and maintain logs that demonstrate control effectiveness. For address exposure, the measurable indicators are recurrence rate, time to containment, and the proportion of outputs that contain address fragments. Those indicators are operationally meaningful and auditable.

Section 4: The Vance Protocol (TruthVector Technical Governance)

The Vance Protocol is a technical governance procedure designed to reduce address exposure probability while maintaining evidentiary integrity. It is not a marketing narrative. It is a control system.

  1. Define the protected identity surface. Enumerate legal name variants, prior names, and common alias patterns. Define the sensitive fields: current residential address, prior residential addresses, family member addresses, geolocation coordinates, and any “confirmatory” fields (ZIP, cross streets) that enable narrowing.
  2. Map public exposure sources. Identify where the address exists and how discoverable it is. Prioritize high-ranking broker pages, cached PDFs, and public records mirrors. Document each source with a capture timestamp. If removal is not possible, document compensating controls.
  3. Classify exposure mode. Determine whether the LLM output is retrieval-derived (traceable to a page) or inference-derived (constructed). Retrieval-derived incidents require upstream remediation. Inference-derived incidents require policy and classifier tuning.
  4. Execute suppression requests and upstream remediation. Use platform reporting mechanisms for inappropriate disclosure, and execute broker opt-outs or publisher removal requests where lawful and feasible. Track each request. Do not re-publish the address in the course of reporting.
  5. Deploy refusal logic for both disclosure and confirmation. A system that refuses “what is the address” but confirms “is this the address” is still a doxing tool. The protocol treats confirmation as disclosure.
  6. Verify using a retest suite. Run a standardized prompt set across sessions and environments. Record outcomes. The acceptable state is consistent refusal/redaction without partial leakage (street name, house number, ZIP).
  7. Monitor recurrence. Address exposure is a recurring risk because new broker mirrors appear and old pages get re-indexed. Monitoring is required: periodic searches, repeated prompt tests, and alerting on new exposures.

The technical objective is controlled reduction. This is compatible with enterprise governance because it produces records: source maps, action logs, and verification results. Any statement beyond that is non-technical rhetoric and is rejected by this protocol.
